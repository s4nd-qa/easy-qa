📄 Виды тестовой документации

🗺️
Тест-план
Главный документ. Стратегия, цель, техники, инструменты, критерии.
📋
Тест-кейс ⭐
Самый частый документ. Подробные шаги для проверки конкретного случая.
✅
Чек-лист
Упрощённый список проверок. Свободный формат.
🐛
Баг-репорт
Отчёт об обнаруженном дефекте: шаги, ОР, ФР, окружение, вложения.

Частая ошибка

Тест-кейс описывает конкретный случай. Нельзя «Проверка корзины» — пиши конкретно: «Добавление товара в корзину авторизованным пользователем».

🐛 Баг-репорт — хороший и плохой

Хороший баг-репорт — тот, по которому разработчик сразу понимает что сломано и как воспроизвести:

Хороший баг-репорт: есть название, шаги, ссылка на запись экрана

А вот пример плохого баг-репорта:

Плохой баг-репорт: нет описания что было сделано до; на записи показан результат, а не как к нему пришли

⚠️
Главная ошибка

Шаги воспроизведения должны быть воспроизводимы любым разработчиком с нуля.
